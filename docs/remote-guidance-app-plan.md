# 手术实时视频远程指导独立应用规划

## 1. 结论

建议做成一个**独立应用**，但不是独立成另一套完全割裂的平台，而是采用：

- 独立前端应用：`apps/remote-guidance`
- 独立实时媒体服务：RTC/录制/弱网处理/监控单独部署
- 复用现有账号、医生认证、后台管理、审计与基础数据

这是当前业务下“稳定性”和“独立性”最平衡的方案。

## 2. 为什么建议独立应用

### 2.1 稳定性

实时音视频属于高敏感、高波动能力，和普通内容页、电商页、后台页的运行特征完全不同：

- 对低延迟、弱网、重连、设备兼容要求高
- 需要长期保活、房间状态同步、录制、媒体监控
- 故障影响直接落到手术中，容错要求远高于普通页面

如果和现有主站强耦合：

- 发布风险会互相传导
- 普通业务改动可能影响实时指导
- 音视频链路的依赖会污染主站

### 2.2 独立性

远程指导后续很可能会长成一条独立产品线：

- 手术远程指导
- 远程病例讨论
- 专家会诊
- 教学直播/示教
- 术后复盘与质控

独立应用更容易形成单独的：

- 域名
- 部署策略
- 监控体系
- 权限模型
- SLA

### 2.3 合规与权限边界

这类能力通常涉及：

- 医生身份验证
- 专家准入
- 房间授权
- 录制留痕
- 审计导出

独立应用更适合把这些高敏感逻辑收在单独边界内，而不是散落到现有多个站点里。

## 3. 不建议的做法

### 3.1 不建议直接塞进现有某个站点

不建议直接做到：

- `apps/cn`
- `apps/admin`
- `apps/www`

原因不是不能做，而是后续一定会出现：

- 依赖越来越重
- 页面职责越来越混
- 发布与回滚越来越难

### 3.2 不建议一开始就完全独立账号体系

不建议把用户、医生、专家、管理员重新做一套。

更好的方式是：

- 复用现有认证
- 复用现有医生审核结果
- 复用现有后台账号与权限
- 只把“实时指导会话域”独立出来

## 4. 推荐架构

## 4.1 应用形态

新增：

- `apps/remote-guidance`

职责：

- 术前会话准备
- 房间加入
- 多角色协同
- 实时视频/语音/消息
- 术中标注与截图
- 录制状态展示
- 术后回放入口

继续复用现有：

- `apps/admin`
  用于排班、配置、审计、录制检索、专家管理
- Supabase
  用于业务数据、权限、审计、会话元数据
- 现有医生认证链路
  用于判断谁可以发起/加入

## 4.2 技术分层

### 业务层

Vetsphere 自己维护：

- 会话预约
- 房间授权
- 角色权限
- 参与者状态
- 术中事件
- 审计日志
- 回放索引

### 媒体层

推荐使用成熟 RTC 平台，例如：

- 首选：LiveKit
- 备选：Daily / Twilio / Agora

媒体层负责：

- WebRTC 房间
- 音视频上下行
- 多端接入
- 录制
- 弱网适配
- 断线重连

### 存储与审计层

继续放在现有体系：

- Supabase 表
- 对象存储
- 后台审计

## 5. 业务边界设计

建议把“远程指导”定义成一个独立业务域，而不是“问诊”的一个字段。

核心对象：

- `guidance_sessions`
- `guidance_participants`
- `guidance_devices`
- `guidance_events`
- `guidance_recordings`
- `guidance_annotations`

这样后面扩展到教学示教、病例讨论时，不需要推翻重来。

## 6. 第一阶段目标

第一阶段不要追求“大而全”，只做最小闭环。

### MVP 范围

- 医生发起远程指导申请
- 指派专家
- 创建会话房间
- 单主画面 + 语音对讲
- 文字消息
- 会话开始/结束
- 基础录制
- 会话审计
- 后台可查询记录

### 暂不进入第一阶段

- AI 语音摘要
- 多机位智能切换
- AR 标注
- 术中自动质控
- 大规模教学直播

## 7. 独立应用与现有系统的关系

### 共享内容

- 用户身份
- 医生认证状态
- 专家身份
- 组织/机构信息
- 审计日志体系

### 独立内容

- 房间服务
- 媒体 token
- 会话状态机
- 录制流水
- 术中操作事件
- 设备接入管理

## 8. 发布与部署建议

建议拆成三层部署：

### 8.1 前端应用

- `apps/remote-guidance`
- 独立域名，例如 `guide.vetsphere.com`

### 8.2 业务 API

- 可继续放在现有 monorepo API 中
- 但建议按模块独立路由：
  - `/api/guidance/sessions`
  - `/api/guidance/tokens`
  - `/api/guidance/recordings`

### 8.3 媒体服务

- 独立部署
- 独立监控
- 独立告警

## 9. 现在就该做的下一步

建议按这个顺序推进：

1. 确认独立应用方案
2. 产出数据模型与角色权限设计
3. 在 monorepo 中新增 `apps/remote-guidance`
4. 接入统一认证与房间 token
5. 搭建最小 RTC 房间
6. 接入后台会话管理

## 10. 我建议的下一步工作内容

如果按这个方向继续，下一步最合适的是：

### A. 先做方案落地设计

输出：

- 数据表设计
- 角色权限矩阵
- 会话状态机
- API 清单
- 页面清单

这是最推荐的下一步。

### B. 直接开始建独立应用骨架

输出：

- 新增 `apps/remote-guidance`
- 基础路由
- 登录校验
- 会话列表页
- 会话详情页
- 房间页骨架

这一步适合在方案设计之后马上做。

## 11. 当前建议

当前建议明确如下：

- 是，建议做独立应用
- 但要复用现有账号、审核、后台和数据底座
- 下一步先做“落地设计”，再开始搭独立应用骨架
